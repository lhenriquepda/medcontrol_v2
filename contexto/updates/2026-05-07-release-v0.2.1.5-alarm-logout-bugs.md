# 2026-05-07 — Release v0.2.1.5 abertura: bugs alarme + logout pré-Reddit

## Contexto

Release anterior v0.2.1.3 (vc 49-51) shipado 2026-05-07 manhã. Pre-Reddit hardening incluiu fix #190 BUG-LOGOUT-RESUME (em `useAppResume.refreshSession()`) que foi validado device user.

Mas no mesmo dia 20:00 BRT, user reportou:
1. **Alarme não tocou** às 20:00 BRT (4 doses pending: Citrato Mg, Gengibre, NAC, Clenil HFA)
2. **App continua deslogando** com frequência mesmo após install vc 50/51 com fix #190

Também doses 19:00 UTC (16:00 BRT) ficaram pending — alarmes pararam de funcionar entre 10:00 e 12:00 BRT no dia.

User pediu pra investigar sem alterar código, abrir release nova pra atacar tudo.

## Investigação (read-only)

Levantamentos via Supabase MCP, código local, Sentry:

1. **2 push_subscriptions com `deviceToken` válido** + 6 stale com NULL
2. **Cron `schedule-alarms-fcm-6h`** rodou OK 18:00 UTC (15:00 BRT) e 00:00 UTC dia seguinte
3. **DB trigger `dose-trigger-handler`** última invocação 14:23 BRT (provavelmente cadastro dose)
4. **prefs:** `criticalAlarm: true`, `push: true`, DND 00:05–07:00 (não bloqueia 20:00)
5. **Sem cron `notify-doses`** — função existe mas SEM schedule no Postgres
6. **`useAuth.jsx:127-143` DELETA push_subscription em SIGNED_OUT** — qualquer SIGNED_OUT (transient ou explícito), incluindo bugs já mapeados em #159/#190
7. **3 reinstalações do APK hoje** (vc 49/50/51) — cada install limpa AlarmManager pending

## Causa raiz combinada

Bug logout (#195/#196) + bug alarme (#197/#198) **se reforçam mutuamente**:

```
SIGNED_OUT spurious (network glitch)
    ↓
useAuth.jsx onAuthStateChange dispara
    ↓
DELETE push_subscription do device  (linha 134-141)  ← BUG #195
    ↓
setUser(null) + qc.clear()                          ← BUG #196 (tela login)
    ↓
Próximo cron schedule-alarms-fcm não envia FCM data pro device
    ↓
AlarmScheduler local não reagenda
    ↓
Alarme não toca (#198 cobre reinstall + #200 cobre sombras gerais)
```

User vê DOIS sintomas relacionados: app desloga + alarme não toca.
