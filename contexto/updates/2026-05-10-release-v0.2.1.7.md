### #204 — Mutation queue offline (Fase 1 offline-first)

**Problema:** mutations (confirmar dose, pular, undo, criar paciente, registrar SOS) tentavam 3 retries com backoff até 30s e descartavam silenciosamente quando offline. App de medicação não pode perder ação crítica do usuário. UI mostrava como confirmado (optimistic update) mas server nunca recebia → dose marcada como pendente no próximo refetch → cálculo da próxima dose contínua errado → healthcare data corrupted silently.

**Implementação:**

1. **`src/services/mutationRegistry.js` novo** — `setMutationDefaults` por chave (12 mutations críticas):
   ```
   ['confirmDose'], ['skipDose'], ['undoDose'], ['registerSos']
   ['createPatient'], ['updatePatient'], ['deletePatient']
   ['createTreatment'], ['updateTreatment'], ['deleteTreatment']
   ['pauseTreatment'], ['resumeTreatment'], ['endTreatment']
   ```
   Cada chave tem `mutationFn` + `onMutate` (optimistic patch + cancelQueries) + `onError` (rollback) + `onSuccess` (track + invalidate) + `onSettled` (refetch debounced 2s).

2. **`src/main.jsx`** atualizações:
   ```js
   defaultOptions: {
     queries:   { networkMode: 'offlineFirst', staleTime: 30_000, ... },
     mutations: { networkMode: 'offlineFirst', retry: 3, ... }
   }
   ```
   Plus bridge `Capacitor.Network.networkStatusChange` ↔ `onlineManager.setOnline()` (fallback `navigator.onLine` + window events na web), `registerMutationDefaults(qc)` chamado ANTES `<PersistQueryClientProvider>` hydrate, `dehydrateOptions.shouldDehydrateMutation: () => true`, `onSuccess: queryClient.resumePausedMutations()`. Buster mantido `v1` (decisão deliberada — bumpar invalidaria cache de TODOS users 1× → pico refetch global; TanStack hydrate é tolerante a campo `mutations` extra).

3. **Hooks refatorados** (`useDoses.js` / `usePatients.js` / `useTreatments.js`) — `useMutation({ mutationKey: ['confirmDose'] })` lookup. Helpers `patchDoseInCache` / `rollback` / `refetchDoses` migrados pra registry centralizado.

4. **`src/components/OfflineBanner.jsx` novo** — banner fixed bottom acima BottomNav. Estados:
   - **Amber:** `!online && pending > 0` → "N ação(ões) salva(s) offline — sincroniza ao reconectar"
   - **Emerald:** `online && wasOffline && pending > 0` por até 3s → "Sincronizando N ação(ões)…"
   - **Hidden:** caso contrário
   - Hooks: `useIsMutating()` + `useOnlineStatus()` + `useEffect` transição.

5. **`src/App.jsx`** — `<OfflineBanner />` integrado pós `<ForceNewPasswordModal>`.

**Auditoria egress completa:**

| Risco | Severidade | Decisão |
|---|---|---|
| `buster v1→v2` invalidaria cache TODOS users 1× | 🔴 Alto pico 1x | **Mantido v1** — TanStack hydrate tolerante a campo extra `mutations` |
| `refetchOnReconnect: true` + bridge Capacitor detecta avião real | 🟡 Médio | **Aceito** — staleTime longo cobre |
| Drain N RPCs em rajada | 🟢 Baixo | **Inevitável** — debounce 2s consolida invalidações |
| `resumePausedMutations()` em todo hydrate | 🟢 Zero | **No-op** se cache vazio |
| Persist mutations localStorage | 🟢 Zero | **Local apenas** |
| `useIsMutating()` em OfflineBanner | 🟢 Zero | **Cache local** |

Net egress incremental ~zero usuários online normais.

**Cobertura mutations:**
- 🔴 Crítica healthcare: confirmDose, skipDose, undoDose, registerSos
- 🟠 Alta: createPatient, updatePatient, deletePatient, createTreatment, updateTreatment, deleteTreatment, pauseTreatment, resumeTreatment, endTreatment
- 🟡 Não-crítica (não cobertas Fase 1): useUserPrefs.update, usePushSubscription.register, sos_rules upsert

**Pendente Fase 1.5:** telemetria PostHog `mutation_queued_offline` + `mutation_drained_online`, `client_request_id` UUID idempotência server-side (item novo backlog).

### #207 — Defesa em profundidade alarme crítico (5 fixes)

**Problema:** user reportou 2026-05-08 19:48 — push FCM 6min antes funcionou mas alarme não disparou. Histórico de inconsistência ("cada hora funciona de um jeito"). Investigação encontrou 4 root causes silenciosos + 1 lacuna de observabilidade.

**Root causes:**

1. **`advanceMins ?? 15` no scheduler** — fallback agendava alarme 15min ANTES do horário se prefs locais não tinham campo explícito. DEFAULT_PREFS sempre declara 0 (alarme exato no horário). Desalinhamento causava alarmes prematuros + confusão.

2. **`SCHEDULE_WINDOW_MS = 48h` (JS) + `HORIZON_HOURS = 72L` (DoseSyncWorker)** — cobertura local insuficiente. User que não abria app por 49h+ ficava sem alarmes locais (dependia FCM cron + Worker, mas Samsung One UI 7 mata Worker periodic).

3. **`firstResetDoneInSession` cache idempotência diff-and-apply (#200.1)** — após primeira execução por sessão, lia `localStorage.dosy_scheduled_groups_v1` pra calcular diff. Se OEM matou AlarmManager mas localStorage continuava dizendo "agendado" → diff vazio → não re-agendava → AlarmManager continuava vazio → alarme NÃO TOCAVA. Drift silencioso.

4. **`REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` ausente** — Samsung One UI 7 + Xiaomi MIUI default colocam apps em bucket "rare/restricted" → kills WorkManager periodic + cancela alarms agendados.

5. **Zero observabilidade prod** sobre quando rescheduleAll roda + quantos alarmes agenda → impossível diagnosticar issues remotamente.

**Fixes:**

1. **`scheduler.js:58 + 244`** — `advanceMins ?? 15` → `?? 0` (alinha DEFAULT_PREFS).

2. **`prefs.js:10`** — `SCHEDULE_WINDOW_MS = 48 * 3600 * 1000` → `168 * 3600 * 1000` (7 dias).
   **`DoseSyncWorker.java:53`** — `HORIZON_HOURS = 72L` → `168L`.

3. **`scheduler.js`** — drop `firstResetDoneInSession` flag + drop `loadScheduledState()` diff logic + drop `toRemove/toUpdate/toKeep` calc + drop `cancelGroup` por-groupId. Substituído por **`cancelAll()` SEMPRE no início + agenda todos do desired set**. Estado persistido apenas pra observabilidade (debug + futuro `getActiveAlarms`).

4. **`AndroidManifest.xml`** — `<uses-permission android:name="android.permission.REQUEST_IGNORE_BATTERY_OPTIMIZATIONS" />`. Play Store policy compliance: medication reminder exception.

   **`CriticalAlarmPlugin.java`** — 3 plugin methods novos:
   - `isIgnoringBatteryOptimizations()` retorna `{ignoring: bool}` via `PowerManager.isIgnoringBatteryOptimizations(packageName)`
   - `requestIgnoreBatteryOptimizations()` abre dialog system `ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` + fallback `ACTION_IGNORE_BATTERY_OPTIMIZATION_SETTINGS`
   - `checkPermissions()` enriquecido com `ignoringBatteryOpt` field; `allGranted` agora inclui essa flag.

   **`criticalAlarm.js`** bridge JS expõe métodos.
   **`PermissionsOnboarding.jsx`** — 5º item "Ignorar otimização de bateria" com action `requestIgnoreBatteryOptimizations` + descrição explicativa Samsung/Xiaomi.

5. **`scheduler.js`** — `Sentry.addBreadcrumb` em `rescheduleAll START` (com `dosesCount` + `patientsCount`) e `END` (com `alarmsScheduled` + `dndSkipped` + `localNotifs` + `summary` + `advanceMins` + `groupsCount`). Crashes futuros virão com trail completo do scheduling state.

**Trade-off scheduling:**

| Mudança | Custo | Justificativa |
|---|---|---|
| Drop diff-and-apply | +200-2000ms janela cancelAll vazia por sessão (mitigada async background, user não percebe) | App de medicação não pode tolerar drift cache vs SO. Custo aceito pra garantia 100% |
| `SCHEDULE_WINDOW_MS` 48h → 168h | ~3x mais alarmes registrados (5 → 28 typical user) | AlarmManager limit ~500/app — folga 20x. Custo Android desprezível |
| `HORIZON_HOURS` Worker 72 → 168 | Worker fetch 2-3x mais rows | Doses ~250 bytes, diferença 10KB/Worker run × 4x/dia = desprezível |
| Sentry breadcrumbs | ~2 breadcrumbs / rescheduleAll (~50/dia user típico) | Quota 1M/mês plano grátis × 100 users = 150k/mês — bem dentro |

**Defense-in-depth resultante (4 caminhos paralelos):**

1. JS-side foreground: rescheduleAll roda toda mudança doses/patients/resume → agenda 7 dias
2. WorkManager 6h periodic: DoseSyncWorker fetch 7 dias futuros + agenda local mesmo app fechado
3. FCM data-driven: cron `schedule-alarms-fcm` envia data message → `DosyMessagingService` agenda local
4. BootReceiver: re-agenda do SharedPreferences pós reboot/upgrade

Battery whitelist (#207.4) garante caminhos 2+3+4 sobrevivem Samsung battery saver.

### CI fix gradlew permission

Investigação CI failure histórico v0.2.1.3 + v0.2.1.4 — `./gradlew: Permission denied` exit 126. Windows commits não preservaram execute bit (`100644` ao invés `100755`).

Fix duplo:
- `git update-index --chmod=+x android/gradlew` → repo agora persiste 100755
- Workflow step "Build signed AAB" agora roda `chmod +x ./gradlew` antes (defesa cobre regressões futuras)

CI v0.2.1.7 ainda falhou com `Tag number over 30 not supported` (keystore JKS legacy incompatível Java 21 ASN.1 parser). User decidiu skipar CI workflow e usar Chrome MCP direto pro upload Play Console (memória chrome_automation reforçada). CI workflow fix pra Java 21 fica como débito — não bloqueador (Chrome MCP cobre fluxo).
