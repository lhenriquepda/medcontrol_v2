## Itens fechados (código)

### 🚀 #205 NOVO P0 — Single source refresh token (storm xx:00 fix)

**Causa raiz:** SQL `auth.refresh_tokens` revelou pattern 100% das storms top-of-hour (xx:00:0X). 20+ refreshes/minuto mesma session (lifespan 16min vs healthy 1-3h). 3 fontes paralelas chamando `/auth/v1/token?grant_type=refresh_token`:
1. JS supabase-js auto-refresh
2. `DoseSyncWorker.refreshAccessToken()` Android WorkManager periodic
3. `DosyMessagingService.refreshAccessToken()` FCM data handler

Race condition `SharedPref.refresh_token` compartilhado + Supabase token reuse detection revogava chain → user re-login forçado 9-12h cycle. Sentry issue `DOSY-P` em v0.2.0.10 confirmou: `Lock "lock:sb-...auth-token" was released because another request stole it`.

**Fix arquitetura:** JS supabase-js é **ÚNICA fonte de refresh**. Native consome `access_token` cached SharedPref atualizado pelo `useAuth.jsx` em SIGNED_IN/TOKEN_REFRESHED/INITIAL_SESSION via plugin `updateAccessToken(accessToken, accessTokenExp)`.

**Arquivos:**
- `android/.../CriticalAlarmPlugin.java`: `setSyncCredentials` aceita `accessToken` + `accessTokenExp` opcionais. Novo `updateAccessToken` method lightweight.
- `android/.../DoseSyncWorker.java`: REMOVE `refreshAccessToken()`. Lê `access_token` cached + verifica exp local margem 60s. Expirado → `Result.success()` skip rodada (next periodic 6h pega token fresco pós JS refresh foreground).
- `android/.../DosyMessagingService.java`: REMOVE static `refreshAccessToken()`. `reportAlarmScheduled` lê access_token cached + skip se expirado (não-crítico — alarme local já scheduled).
- `src/services/criticalAlarm.js`: setSyncCredentials payload inclui accessToken+accessTokenExp. Novo `updateAccessToken({})` export.
- `src/hooks/useAuth.jsx`: listener propaga `s.access_token` + `s.expires_at * 1000` via setSyncCredentials.

**Trade-offs:**
- WorkManager rodada pode skip se app não foreground >1h após token expirar — AlarmManager local da rodada anterior cobre HORIZON 168h.
- `reportAlarmScheduled` skip silent → cron notify-doses fallback push tray cobre.

### 🚀 #204 expand — Fixes A1/A2/B/C (logcat-driven)

**Fix B (pre-mount Network.getStatus bloqueante):** `await Network.getStatus()` em `boot()` async pré-React mount. Sem isso, mutations rehydradas em avião mode tentavam executar antes Capacitor confirmar offline → 1s race → failureCount inflado → re-pause.

**Fix C (onlineManager.setEventListener):** Capacitor.Network única fonte de verdade. Substitui default subscriber TanStack que disparava `setOnline` espúrio via `window.online`/`offline` events em Capacitor WebView Android. Logcat v0.2.1.7 mostrou caller `vendor-data-*.js` flipando 7ms antes Capacitor confirmar.

**Fix A1 (resolve temp patientId):** createPatient onSuccess marca real patient com `_tempIdSource: tempId` no cache. createTreatment mutationFn resolve `patientId.startsWith('temp-')` → lookup `_tempIdSource` → substitui real ID antes RPC. Antes drain pós-reconnect retornava FK violation → failureCount=4 status=error → mutation descartada.

**Fix A2 (doses optimistic local):** createTreatment onMutate gera doses optimistic via `generateDoses` util → insere cache `['doses']` todas variações queryKey. Dashboard renderiza tratamento offline + `App.jsx:182 scheduleDoses(allDoses)` triggera AlarmScheduler → alarme nativo agenda offline → AlarmReceiver dispara no horário (validado device).

### 🚀 Optimistic CRUD completos

`updatePatient` + `updateTreatment` + `pauseTreatment` + `resumeTreatment` + `endTreatment` + `registerSos` cada com onMutate cache patch + onError snapshot rollback + onSuccess invalidate. Plus helper `patchEntityListsInCache` varre `findAll({queryKey: [entity]})` + setQueryData em CADA variação `[entity, filter]` (fix bug: `setQueryData(['treatments'])` exato não atingia `['treatments', {patientId}]`).

### 🚀 Forms edit path offline

`PatientForm.jsx` + `TreatmentForm.jsx` detectam `onlineManager.isOnline() === false` em CREATE + EDIT paths → `mutate(payload)` fire-and-forget + toast claro "Salvo/Alterações salvas offline" + close modal imediato. Antes: `mutateAsync` esperava reconnect → modal travava em loading.

### 🚀 useOfflineGuard + OfflineNotice (avisos UX honestos)

- `src/hooks/useOfflineGuard.js` (novo): `guard.ensure(label)` bloqueia + toast "Sem conexão. {feature} requer internet." + retorna boolean. Plus `online` state reativo pra disabled UI.
- `src/components/OfflineNotice.jsx` (novo): banner contextual reusable (variant inline/compact). Render null se online.
- `SharePatientSheet.jsx`: guard share/unshare + button disable + banner.
- `SOS.jsx`: guard saveRule (SOS rules fora queue) + queue registerSos offline-aware + banner.
- `Settings/index.jsx`: guard exportar LGPD + excluir conta + banner topo.
- `TreatmentForm.jsx`: saveAsTemplate offline → toast warn "Salvar modelo requer internet — refaça online".

### 🐛 Bug fix usePatient/useTreatment offline fallback

`PatientDetail` travava em "Carregando…" infinito quando user navegava direto pra detalhe offline. Cache `['patients', id]` vazio + query falhou → `patient=undefined`. Fix: `initialData` lookup na lista `['patients']` cache (PATIENT_COLS_LIST tem dados suficientes; photo_url full vem do fetch online).

`useTreatments({patientId})` mesmo problema — `setQueryData(['treatments'])` exato não atingia `['treatments', {patientId}]`. Fix: varre todas queries `['treatments', *]` + dedup por id + filter client-side.

`useTreatment(id)` análogo — `findAll` varre todas variações + extrai por id.

## Decisões arquiteturais

1. **JS supabase-js = única fonte refresh** (#205). Native consome cached. Trade-off: rodada Worker eventualmente skip se app não foreground por >1h.
2. **initialData lookup pattern offline** — useQuery resolve `['entity', filter]` via cache lista geral `['entity']` qualquer filter. Reusable em outros hooks. Aplicado: usePatient, useTreatment, useTreatments(filter).
3. **patchEntityListsInCache helper** — setQueryData em TODAS variações queryKey `[entity, *]`. Substitui padrão antigo `setQueryData(['entity'])` queryKey exata.
4. **useOfflineGuard + OfflineNotice UX honesto** — features fora queue (LGPD/shares/SOS rules/templates) bloqueiam + toast claro em vez de iludir que "vai sincronizar quando reconectar".
