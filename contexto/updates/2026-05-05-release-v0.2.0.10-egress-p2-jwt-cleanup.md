## Items fechados v0.2.0.10

### #139 — dose-trigger-handler skip se scheduledAt > 6h

`supabase/functions/dose-trigger-handler/index.ts` v11. Trigger DB dispara webhook por INSERT em `doses`. Antes mandava FCM data pra TODA dose criada, mesmo doses agendadas pra dias/semanas a frente.

Agora: filter `scheduledAt > now + 6h` → return early `skipped: 'beyond-cron-horizon'`. Doses dentro 6h continuam fluxo normal (webhook → FCM data → AlarmManager). Doses > 6h serão pegas pelo cron `schedule-alarms-fcm` (rodando a cada 6h, 24h horizon após #140).

```ts
const SIX_HOURS_MS = 6 * 60 * 60 * 1000
if (scheduledAt.getTime() > Date.now() + SIX_HOURS_MS) {
  return new Response(JSON.stringify({ ok: true, skipped: 'beyond-cron-horizon' }))
}
```

**Estimado: -50% a -70% chamadas Edge function** (cadastro batch tratamento contínuo gera 30-90 doses futuras → trigger só fire pra próximas 6h).

### #140 — schedule-alarms-fcm HORIZON 72h → 24h

`supabase/functions/schedule-alarms-fcm/index.ts` v10. Cron `pg_cron` roda a cada 6h, antes pegava doses próximas 72h. AlarmManager nativo Android re-agenda a cada call → 24h coverage com folga 4× (4 ciclos × 6h = 24h).

Payload FCM ~3× menor por device (24h doses vs 72h). Multi-tenant: 100 users × ~10 doses/dia × FCM data = ~1000 mensagens FCM/24h vs ~3000/72h. Cleanup automático.

**Estimado: -65% payload FCM data + -40% egress Edge function chain**.

### #141 — useReceivedShares staleTime 60s → 5min

`src/hooks/useShares.js`. Hook lista shares recebidos pelo user (paciente compartilhado por outro). Antes refetchava a cada 60s. Realidade: shares mudam raríssimo (usuário aceita 1× e fica). 5min suficiente.

```js
return useQuery({
  queryKey: ['shares-received', user?.id],
  staleTime: 5 * 60 * 1000, // era 60_000
})
```

**Estimado: -80% calls listReceivedShares**.

### #143 — useUserPrefs getSession() vs getUser()

`src/hooks/useUserPrefs.js`. Antes hook chamava `supabase.auth.getUser()` — round-trip /auth/v1/user pra validar JWT no server. Plus useUserPrefs invalida em vários paths (resume, realtime, navigation). Cada call = 1 request /auth/v1/user.

Agora: `supabase.auth.getSession()` — local-only, lê localStorage cache (Supabase JS já valida exp client-side). Sem round-trip.

```js
const { data: { session } } = await supabase.auth.getSession()
const user = session?.user
```

**Estimado: -100% calls /auth/v1/user em useUserPrefs path** (que era um dos top paths em GET volume).

### #142 — JWT cron cleanup cosmético

Após verificação #142 v0.2.0.9 (Legacy JWT secret revoked → PostgREST 401 com JWT antigo), JWT hardcoded `iat: 26 abr` no cron job 3 era safe mas ainda apareciam suspeitos em audits ou logs Supabase. Cleanup:

1. `SELECT cron.unschedule('schedule-alarms-fcm-cron')` (drop antigo)
2. Recreate sem header `Authorization` — Edge function é pública (`verify_jwt: false`), authorize via env `SERVICE_ROLE_KEY` interna
3. `INSERT INTO cron.job` com `command` chamando `net.http_post` sem header JWT

Resultado: cron limpo, sem JWT em código DB. Verificado funcional via logs Supabase Edge function (continua rodando 6h em 6h).

### #147 — BUG-041 reformulação fluxo recuperação senha (catalogado)

User reportou tentando recuperar senha esposa Daffiny → Supabase email recovery link aponta `localhost:3000/recover` → quebrado em prod (Vercel `dosymed.app` é prod URL).

Workaround imediato: SQL direto reset senha Daffiny pra `123456`:
```sql
UPDATE auth.users
SET encrypted_password = crypt('123456', gen_salt('bf'))
WHERE email = 'daffiny@dafhost.com.br';
```

Catalogado #147 BUG-041 P1 com plano completo:
- Reformular Recovery flow: Supabase config Site URL + redirect URLs allowlist
- AuthContext novo route `/auth/recover` consome token
- Force HTTPS dosymed.app em links (não localhost)
- E2E test recovery em prod
