## Items fechados (código)

### #092 BUG-025 [P0 CRÍTICO financial] — Egress 400% Free Plan reduction

**Problema:** Egress Supabase 20GB/5GB Free Plan (400%) com apenas 2 users cadastrados. Insustentável mesmo pago em escala (10k users projetariam consumo massivo).

**Causa raiz multi-frente:**

1. **Realtime postgres_changes sem filter** — channel subscribe a 7 tabelas (`patients`, `treatments`, `doses`, `sos_rules`, `treatment_templates`, `subscriptions`, `patient_shares`) sem filtro server-side. Cada change em qualquer row de qualquer user era broadcast tentado pra TODOS clients conectados (RLS bloqueia leitura mas mensagem websocket tráfega).
2. **listDoses sem range default** — `useDoses()` em App.jsx chamado sem from/to puxa histórico inteiro (5+ anos × 4 doses/dia ~7000 rows × 250 bytes = 1.75 MB / refetch).
3. **listDoses paginate cap 20 pages** — até 20k rows por refetch.
4. **queryKey timestamps instáveis** — callers passam `new Date().toISOString()` inline em filter.from/to → queryKey diferente cada render → refetch storm.
5. **refetchInterval 60s + staleTime 30s** — agressivo demais. Realtime já cobre updates instantes.
6. **App.jsx useDoses() sem filter** — alarm reschedule pulls histórico todo (mas só precisa futuros 14d).

**Fix aplicado:**

Cliente (`src/`):
- `useRealtime.js`:
  - postgres_changes filter server-side `userId=eq.${user.id}` em todas tabelas com userId column
  - `subscriptions` removido do TABLE_TO_KEYS (admin-only, raras mudanças)
  - invalidateQueries scoped por table (era invalidate ALL queries)
- `dosesService.js` `listDoses`:
  - Default range fail-safe (-30d / +60d) se caller não passar from/to
  - Paginate cap 20 → 5 pages (5k rows max)
- `useDoses.js`:
  - queryKey normaliza timestamps pra hour boundary (estável dentro da hora corrente)
  - refetchInterval 60s → 5min
  - staleTime 30s → 2min
  - refetchOnMount: 'always' → true (respeita staleTime)
- `App.jsx`:
  - alarm reschedule scope explícito -1d / +14d (era pull histórico todo)
  - useMemo + hourTick pra queryKey estável
- staleTime bump em hooks rarely-mutated:
  - `useUserPrefs` 30s → 10min
  - `usePatients` 6s → 5min
  - `useTreatments` 6s → 5min
  - `useMyTier` 60s → 30min

**Path crítico (alarme/push) NÃO regrediu:**
- `dose-trigger-handler` (INSERT trigger) intacto
- `schedule-alarms-fcm` (cron 6h) intacto
- `notify-doses` (cron) intacto
- AlarmManager nativo (Android) recebe FCM data via servidor, independente de TanStack cache cliente

### #093 BUG-026 [P1] — Race condition useRealtime

**Problema:** Sentry alert "Crash spike >10/h" disparou. Erro `Vr.on` (Supabase Realtime client): "cannot add postgres_changes callbacks after subscribe()". 23 events 24h / 57 events 90d / 3 users.

**Causa raiz:** `unsubscribe()` chamava `supabase.removeChannel()` async sem await. `subscribe()` recriava channel com mesmo nome `realtime:${userId}` antes do anterior terminar removal → Supabase reusava channel cached ainda em estado SUBSCRIBED → `.on()` throw.

Triggers do reconnect: TOKEN_REFRESHED listener (#077), watchdog 60s, native pause/resume, onStatusChange backoff retry.

**Fix aplicado em `useRealtime.js`:**
1. Channel name único por subscribe: `realtime:${userId}:${gen}:${Date.now()}`
2. `await supabase.removeChannel(ch)` em unsubscribe (era fire-and-forget)
3. Generation counter (`myGen !== generation`) ignora callbacks de canais antigos durante reconnect

## Estimativa de redução egress

Cenário típico (1 user idle no Início, app aberto 1h):
- **Antes:** 60 refetches × 1.75MB (histórico todo) = 105MB/h × 24h = 2.5GB/user/dia
- **Depois:** 12 refetches × ~50KB (90d window scope) = 600KB/h × 24h = 14MB/user/dia
- **Redução estimada:** ~99% pra cenário client refetch
- **Realtime broadcasts:** filter server-side reduz ~90% mensagens (10k users × N changes outros vs próprio user)

Meta: <5GB/mês com 2 users (tem que validar). Provavelmente seguro até dezenas de users sem upgrade Pro.
