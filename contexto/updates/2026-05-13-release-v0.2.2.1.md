## Motivação

Audit log v0.2.2.0 imediato pós-deploy revelou comportamento inesperado: rescheduleAll JS rodando 1×/min recorrente. 868 rows audit em 30min teste (esperado ~10). Storm silencioso pré-existente — sempre existiu, era invisível antes da observabilidade.

## Root causes (3 simultâneos)

### 1. Storm 1/min
App.jsx useEffect (`scheduleDoses(allDoses, {patients: allPatients})`) deps array `[user, allDoses, allPatients, dosesLoaded, patientsLoaded, scheduleDoses]`. Algum dep flipa ref a cada 60s. Suspeita primária: realtime channel reconnect (watchdog 60s em useRealtime.js) → refetchQueries(['doses']) → new data ref → useEffect re-fires → rescheduleAll. Investigação root cause = item separado #212. Throttle resolve sintoma sem perder updates real.

### 2. Window 168h em vez de 48h
`prefs.js SCHEDULE_WINDOW_MS = 168 * 3600 * 1000` (7 dias) — comentário inline dizia "48h" mas valor era 168h. Plan #209 alinhamento com daily-alarm-sync 5am + Worker 6h era 48h. 168h × 14-30 doses/dia = ~100-200 doses agendadas/batch desnecessariamente.

### 3. Audit per-group inserts
scheduler.js wire v0.2.2.0 chamava `logAuditEventsBatch(group.map(...))` per iteration. 10 grupos × 1 insert = 10 separate INSERTs/batch. Storm × N inserts = degradação egress + latência.

## Implementação

### Fix A: Throttle module-level rescheduleAll (30s + trailing run)

`src/services/notifications/scheduler.js`:

```js
const RESCHEDULE_THROTTLE_MS = 30_000
let _lastRunAt = 0
let _pendingTrailing = null
let _pendingArgs = null

export async function rescheduleAll(args = {}) {
  if (!isNative) return rescheduleAllWeb(args.doses, args.patients, args.prefsOverride)
  const now = Date.now()
  const elapsed = now - _lastRunAt
  if (elapsed < RESCHEDULE_THROTTLE_MS) {
    _pendingArgs = args
    if (!_pendingTrailing) {
      _pendingTrailing = setTimeout(() => {
        _pendingTrailing = null
        const a = _pendingArgs; _pendingArgs = null
        rescheduleAll(a)
      }, RESCHEDULE_THROTTLE_MS - elapsed)
    }
    return { throttled: true }
  }
  _lastRunAt = now
  if (_pendingTrailing) { clearTimeout(_pendingTrailing); _pendingTrailing = null; _pendingArgs = null }
  return _rescheduleAllImpl(args)
}
```

Pattern: primeira execução roda imediato. Requests dentro janela 30s coalescem em single trailing run com last args. Garante updates real (dose confirm/skip/undo invoke scheduleDoses via mutationRegistry) NÃO são perdidas — apenas atrasadas até 30s.

### Fix B: Window 168h → 48h

`src/services/notifications/prefs.js`:

```js
export const SCHEDULE_WINDOW_MS = 48 * 3600 * 1000 // 48h
```

Alinha plan #209: daily-alarm-sync cron 5am BRT (48h horizon) + Worker periodic 6h (48h horizon) + dose-trigger real-time mantêm janela rolling. Window 48h = ~30 doses/batch normal.

### Fix C: Audit accumulator + single batch insert

`src/services/notifications/scheduler.js`:

```js
const auditAccumulator = []
auditAccumulator.push({ action: 'batch_start', metadata: {...} })
// ... per group iteration: auditAccumulator.push(...)
auditAccumulator.push({ action: 'batch_end', metadata: {...} })
logAuditEventsBatch(auditAccumulator)
```

Cobre 3 return paths: nothing_to_schedule (early return), error (LocalNotifications.schedule fail), normal completion. 1 insert/batch com batch_start + N scheduled + batch_end.

### Fix D: DB GRANTs (descoberto durante debug)

Migrations aplicadas:
- `grant_service_role_audit_tables`
- `grant_authenticated_audit_tables`

```sql
GRANT SELECT, INSERT, UPDATE, DELETE ON medcontrol.alarm_audit_log TO service_role;
GRANT SELECT, INSERT, UPDATE, DELETE ON medcontrol.alarm_audit_config TO service_role;
GRANT USAGE ON SCHEMA medcontrol TO authenticated, anon;
GRANT INSERT, SELECT ON medcontrol.alarm_audit_log TO authenticated;
GRANT SELECT ON medcontrol.alarm_audit_config TO authenticated;
```

Bug raiz audit v0.2.2.0: tabelas criadas com RLS policies OK, mas table-level GRANTs faltando. Edge Functions (service_role) e user client (authenticated) silenciavam INSERTs (RLS check nem chegava a executar — permission denied antes).
