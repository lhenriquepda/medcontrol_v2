### Bloco 2 — Fix B-01 + A-03 (commit `f8596c7`)
- **B-01:** `AlarmReceiver.onReceive` agora cancela PendingIntent tray pendente AlarmManager (`AlarmManager.cancel(trayPi)`), não apenas notif visível (`NotificationManagerCompat.cancel`). Resolve race entre AlarmReceiver + TrayNotificationReceiver mesmo timestamp
- **A-03:** Snooze persist via novo `AlarmScheduler.persistSnoozedAlarm` (wrapper público de `persistAlarm`). Chamado por `AlarmActionReceiver.scheduleSnooze` + `AlarmActivity.scheduleSnooze`. Reboot dentro 10min mantém triggerAt snoozed correto (BootReceiver lê dali)

### Bloco 3 — Plano A unifica tray Java M2 (commit `88d7f17`)
- **AlarmScheduler.java:** `scheduleTrayGroup` public method + `persistTrayEntry` + `loadPersistedTrayEntries` + `saveTrayEntries` + `removePersistedTrayEntry` + `cancelTrayGroup` + `cancelAllTrays`
- **Namespace `dosy_tray_scheduled` SharedPreferences** persiste tray entries pra BootReceiver re-agendar
- **CriticalAlarmPlugin:** `@PluginMethod scheduleTrayGroup`, `cancelTrayGroup`, `cancelAllTrays`
- **criticalAlarm.js:** bridges JS `scheduleTrayGroup`, `cancelTrayGroup`, `cancelAllTrays`
- **scheduler.js:** rescheduleAll refactor — branches usam `scheduleTrayGroup` (Java) em vez de `LocalNotifications.schedule` (Capacitor). Daily summary continua Capacitor (caso especial repeat=day)
- **channels.js cancelAll:** cobre 3 fontes (cancelAllCriticalAlarms + cancelAllTrays + LocalNotifications cleanup)
- **BootReceiver:** re-agenda tray entries persistidas pós reboot (antes só re-agendava alarmes nativos)

### Bloco 4 — Fix B re-rota fire time (commit `c8554c3`)
- `AlarmReceiver.onReceive` consulta SharedPrefs `dosy_user_prefs` ANTES de fire AlarmService

- Branch dinâmico no fire time — doses agendadas via FCM/Worker com prefs antigas respeitam prefs atuais
- `AlarmScheduler.isInDndWindowPublic` exposto pra AlarmReceiver

### Bloco 5 — Fix C + A-02 batch cancel (commit `0bb8070`)
- **SQL migration `dose_change_batch_trigger_v0_2_3_1`:**

  - Trigger AFTER DELETE FOR EACH STATEMENT (novo, batch)

- **Migration `add_cancelled_status_to_doses_v0_2_3_1`:** CHECK constraint aceita `'cancelled'`
- **Edge dose-trigger-handler v20 deployed:** `BATCH_UPDATE`/`BATCH_DELETE` agrupa old_rows por (ownerId, patientId) + 1 FCM por device com CSV completo
- **treatmentsService.cancelFutureDoses:** DELETE → UPDATE status='cancelled' (preserva histórico + dispara batch trigger)
- **DosyMessagingService.handleCancelAlarms:**
  1. Cancel cada doseId individualmente (single-dose groups, hash bate)
  2. **Fix C:** reconstroi hash `sortedDoseIds.join('|')` e cancela tambem (cobre multi-dose groups)
